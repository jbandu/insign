- **Duration:** 2 weeks
- **Team Size:** 3-5 developers

- **Review Meeting:** Friday, Sprint End (1 hour)
- **Retrospective:** Friday, Sprint End (1 hour)

---

## ðŸŽ¯ Sprint Planning Process

### 1. Pre-Planning (Day before)
- [ ] Product Owner reviews and prioritizes backlog
- [ ] Team reviews upcoming stories
- [ ] Dependencies identified
- [ ] Questions documented

### 2. Sprint Planning Meeting
1. **Review Sprint Goal** (15 min)
   - What is the primary objective?
   - What value will we deliver?

2. **Story Selection** (60 min)
   - Pull stories from backlog
   - Ensure dependencies are met
   - Confirm story points
   - Assign stories to developers

3. **Task Breakdown** (30 min)
   - Break stories into tasks (<4 hours each)
   - Identify technical risks
   - Plan testing approach

4. **Capacity Planning** (15 min)
   - Account for: holidays, PTO, meetings
   - Adjust commitment if needed

### 3. During Sprint
- [ ] Daily standups (15 min)
- [ ] Update story status daily
- [ ] Unblock issues quickly
- [ ] Maintain sprint board

### 4. Sprint Review
- [ ] Demo completed stories
- [ ] Gather feedback
- [ ] Update product backlog

### 5. Sprint Retrospective
- [ ] What went well?
- [ ] What didn't go well?
- [ ] Action items for next sprint

## âœ… Definition of Ready (DoR)

Before a story enters a sprint, it must have:
- [ ] Clear acceptance criteria
- [ ] Story points estimated
- [ ] Dependencies identified
- [ ] Technical approach discussed
- [ ] UI/UX mockups (if needed)
- [ ] Test scenarios defined

---

## âœ… Definition of Done (DoD)

Before a story is marked complete:
- [ ] Code implemented per acceptance criteria
- [ ] Peer reviewed and approved
- [ ] Unit tests written (>80% coverage)
- [ ] Integration tests passed
- [ ] Documentation updated
- [ ] Deployed to staging
- [ ] QA tested
- [ ] Product Owner accepted
- [ ] Security review (if applicable)

## ðŸš§ Risk Management

### Common Risks
1. **Scope Creep**
   - Mitigation: Strict adherence to sprint commitment
   - Action: New requirements go to backlog

2. **Dependencies**
   - Mitigation: Identify early, plan around
   - Action: Daily check-in on blocking items

3. **Technical Unknowns**
   - Mitigation: Spike stories for research
   - Action: Time-box investigations

4. **Resource Availability**
   - Mitigation: Plan for PTO and holidays
   - Action: Adjust capacity accordingly

## ðŸŽ¯ Sprint Board States

### Columns
1. **To Do** - Story committed but not started
2. **In Progress** - Actively being worked on
3. **Code Review** - PR submitted, awaiting review
4. **Testing** - In QA testing
5. **Done** - Meets DoD, ready for deployment

### WIP Limits
- In Progress: 1 story per developer
- Code Review: Max 3 PRs
- Testing: Max 2 stories
